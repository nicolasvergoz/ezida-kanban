## Why

The landing page shows a screenshot of the viewer; visitors cannot try the product without installing it. A static, in-memory demo of the viewer at `/demo/` lets prospects feel the UX directly — drag cards, edit titles, rename columns, switch theme — using Ezida's own `kanban.toml` as fixture content. Because GitHub Pages is static, mutations must be in-memory only and reset on refresh; the source board snapshot is regenerated on every deploy so the demo always reflects the current state of the project's own board.

## What Changes

- Add `ezida export --json` command that emits the same JSON envelope as `GET /api/board` (schema_version, project_name, columns, priorities, cards_per_column, cards) for one-shot snapshotting.
- Add `site/demo/` directory containing a thin wrapper over the viewer:
  - `site/demo/index.html` — copy of the viewer's `index.html` with an additional `<script src="demo-shim.js">` (loaded before `app.js`) and a "Demo — snapshot &lt;sha&gt; · changes don't persist" banner.
  - `site/demo/demo-shim.js` — monkey-patches `window.fetch` and `window.EventSource` to route `/api/*` traffic to an in-memory state initialised from `board.json`.
  - Symlinks `site/demo/app.js`, `site/demo/style.css`, `site/demo/vendor` pointing into `internal/server/web/` so the demo stays byte-identical to the real viewer with no duplication.
  - `site/demo/board.json` — generated by CI, gitignored. Snapshot of Ezida's own board at deploy time.
- Extend `.github/workflows/pages.yml`:
  - Add `kanban.toml` to the `paths` trigger so a board-only commit redeploys.
  - Build `ezida` with `go build` then run `ezida export --json > site/demo/board.json` before uploading the Pages artifact.
  - Substitute the snapshot SHA into the demo banner from `$GITHUB_SHA`.
- Add `site/demo/board.json` to `.gitignore`.
- Add a "Live demo →" link to the `#webui` and `#roadmap` sections of `site/index.html`.

## Capabilities

### New Capabilities
- `demo-viewer`: requirements for the static, in-memory demo of the viewer hosted at `/demo/` on the landing page.
- `board-export`: requirements for the `ezida export --json` CLI command.

### Modified Capabilities
- `site-deployment`: pages workflow now triggers on `kanban.toml` changes and runs `ezida export` before deploy.

## Impact

- `cmd/ezida/main.go` — register new `export` command.
- `internal/commands/export.go` (new) + `export_test.go`.
- `site/demo/` — new directory with HTML, shim JS, symlinks.
- `site/index.html` — two new links.
- `.github/workflows/pages.yml` — extended `paths`, new Go build + export steps.
- `.gitignore` — one new entry.
- No changes to the Go server, board package, or existing viewer assets.
